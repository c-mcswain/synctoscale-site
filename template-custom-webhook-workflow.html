Custom Webhook → Workflow

Template

Connect any system that can send HTTP requests to SyncToScale. Define a custom webhook endpoint, map incoming JSON, and orchestrate downstream actions — with Nova helping validate payloads and suggest mappings.

Category: Developer & Custom Difficulty: Advanced Trigger: Incoming webhook Systems: Any system · SyncToScale · Nova

Overview

The Custom Webhook → Workflow template lets you integrate tools that don’t have a native connector (yet). You define a JSON payload shape, receive events over HTTPS, and use SyncToScale to map fields, branch logic, and trigger actions in the rest of your stack — while Nova inspects examples, suggests mappings, and validates payloads in real time.

When to use this template

  • You have an internal tool that can call webhooks, but no API connector.
  • You want to prototype a new integration quickly.
  • You need a flexible “catch-all” endpoint for events.
  • You want Nova to help design mappings from sample JSON.

What this template automates

  • Creates a secure, unique webhook URL for your workflow.
  • Normalizes and validates incoming JSON payloads.
  • Maps payload fields into SyncToScale objects.
  • Routes events to actions (CRM, HRIS, ticketing, Slack, etc.).
  • Logs events and Nova diagnostics for debugging.

How it works

  1. You create a Custom Webhook template instance in SyncToScale.
  2. SyncToScale generates a unique HTTPS endpoint and secret.
  3. Your external system sends JSON payloads to that endpoint.
  4. Nova analyzes examples and proposes field mappings and validations.
  5. Mapped data is passed into your downstream workflow steps.
  6. All runs are logged with payload snapshots and Nova insights.

Workflow steps

  1. Step 1 — Trigger: Incoming HTTP webhook

    The workflow begins when an HTTP POST request hits your unique SyncToScale webhook URL with a JSON body. Optional HMAC signatures or bearer tokens can be required.

  2. Step 2 — Validate & parse JSON

    SyncToScale checks that:

    • The body is valid JSON.
    • Required keys are present.
    • Optional signature headers match the expected secret (if configured).

    Nova assists by:

    • Generating a proposed JSON schema from example payloads.
    • Highlighting fields that are often missing or inconsistent.
    • Suggesting which fields should be required or optional.

  3. Step 3 — Transform & map fields

    You configure mappings from arbitrary JSON keys into structured fields (like account, user, ticket, or event). Nova can suggest mapping rules based on names and sample values.

  4. Step 4 — Route to downstream actions

    Based on payload content (event type, severity, source, etc.), the workflow branches into actions like:

    • Create or update records in other systems.
    • Send Slack alerts or emails.
    • Append events into a log or analytics store.

  5. Step 5 — Respond to the webhook caller

    SyncToScale returns a configurable HTTP status and optional JSON response indicating success, partial success, or error details.

Field mappings

Webhook JSON → SyncToScale event

Incoming JSON key Transform Mapped field
event_type Lowercase eventType
source_system Trim + lowercase source
timestamp Parse to ISO-8601 occurredAt
payload.user.email Lowercase & validate userEmail
payload.account_id Stringify accountExternalId

Routing rules

Condition Action Example
eventType = "ticket.created" Create CRM case + Slack alert Support form submission
eventType = "usage.spike" Post to product-ops channel Usage anomaly notifications
eventType = "billing.charge_failed" Create billing task + email CSM Billing system webhook

Sample JSON payloads

Example incoming webhook payload

{
  "event_type": "ticket.created",
  "source_system": "internal_help_portal",
  "timestamp": "2025-04-03T16:20:00Z",
  "payload": {
    "ticket_id": "TCK-10291",
    "subject": "Cannot access reporting dashboard",
    "priority": "high",
    "user": {
      "email": "morgan.shaw@demo-company.com",
      "name": "Morgan Shaw"
    },
    "metadata": {
      "plan": "Enterprise",
      "account_id": "ACCT-4390"
    }
  }
}

Internal normalized event (after mapping)

{
  "eventId": "evt_01J2WEBHOOK",
  "eventType": "ticket.created",
  "source": "internal_help_portal",
  "occurredAt": "2025-04-03T16:20:00Z",
  "accountExternalId": "ACCT-4390",

  "data": {
    "ticketId": "TCK-10291",
    "subject": "Cannot access reporting dashboard",

Response sent back to caller

{
  "status": "accepted",

  "novaSummary": "Ticket created in Zendesk and Slack alert sent to #support-sev-high."
}

Error handling

Because external systems rely on HTTP responses, this template includes explicit, predictable error semantics.

  • 400 Bad Request for invalid JSON or missing required keys.
  • 401/403 for invalid or missing authentication signatures.
  • 202 Accepted for async processing even when downstream systems are busy.
  • All failures are logged with payload snippet and Nova diagnostic notes.

Sample error diagnostic

{
  "runId": "run_01J2WEBHOOKFLOW",
  "status": "failed",
  "httpStatus": 400,
  "error": {
    "type": "schema_validation_failed",
    "message": "Required field 'event_type' missing from payload",
    "novaHints": [
      "Check the integration docs for required keys.",
      "Example payloads suggest event_type is usually present."
    ]
  }
}

Nova insights for this template

  • Suggests mappings from sample payloads to SyncToScale fields.
  • Identifies common payload errors and missing fields across runs.
  • Groups similar events to propose reusable workflow branches.
  • Highlights external systems with high failure rates or poor payload hygiene.

Set up this template in SyncToScale

  1. Select Custom Webhook → Workflow from the Templates library.
  2. Copy the generated webhook URL and secret into your external system.
  3. Send a few sample payloads in “Preview” mode.
  4. Use Nova’s suggestions to define required fields and mappings.
  5. Add downstream actions (Slack, CRM, ticketing, etc.).
  6. Switch from Preview to Active once validated.

Best practice defaults

  • Enable HMAC or token-based authentication on the webhook.
  • Use Nova to lock down a JSON schema before going live.
  • Return 202 for long-running workflows rather than 200.

Nova will help you…

  • Design your payload schema from real examples.
  • Catch malformed payloads before they break flows.
  • Discover patterns across events and propose re-usable templates.
