Field Mapping

Field mapping transforms incoming data into the structure required by each system. SyncToScale supports direct mapping, formulas, conditional logic, schema normalization, and Nova-assisted intelligent mapping.

Overview

When a trigger fires, the incoming payload is rarely in the exact format another system expects. Field mapping ensures objects are aligned across applications by handling conversions, transformations, renaming, and schema differences.

Direct Field Mapping

The simplest form of mapping: passing values from source → destination without modification. 

{
  "mapping": {
    "first_name": "payload.employee.firstName",
    "last_name": "payload.employee.lastName",
    "email": "payload.employee.workEmail"
  }
}

Missing fields can be defaulted or flagged by the validator.

Schema Normalization

Systems often use different object shapes for similar data. SyncToScale normalizes common objects (employee, user, contact, ticket) to a consistent schema. 

Incoming (HRIS):
{
  "fname": "Jordan",
  "lname": "Lee",
  "mail": "jordan@example.com"
}

Normalized:
{
  "first_name": "Jordan",
  "last_name": "Lee",
  "email": "jordan@example.com"
}

This allows downstream systems to expect predictable field names.

Transformation Mapping

Transformations alter or combine values using formulas. 

{
  "mapping": {
    "full_name": "payload.first_name + ' ' + payload.last_name",
    "start_date_iso": "formatDate(payload.startDate, 'YYYY-MM-DD')"
  }
}

Nova can automatically detect formatting mismatches and recommend corrections.

Conditional Mapping

Conditional logic allows fields to change based on attributes in the payload. 

{
  "mapping": {
    "employment_type": "payload.isContractor ? 'contractor' : 'full_time'",
    "department": "payload.department || 'General'"
  }
}
  • Useful when upstream systems do not consistently populate optional fields.
  • Prevents workflow failures due to null or undefined values.

Lookup-Based Mapping

SyncToScale can map fields based on reference values such as IDs, group names, or role mappings. 

{
  "mapping": {
    "role_id": "lookup('roles', payload.jobTitle)"
  }
}

Lookups help translate between systems with mismatched internal identifiers.

Nova Intelligent Mapping

Nova analyzes naming patterns, historical mappings, and payload structures to:

  • Suggest likely field matches
  • Auto-map common object types
  • Flag incompatible data types
  • Recommend transformations
  • Detect missing required fields
Nova Suggestion:
{
  "email": "payload.workEmail",
  "first_name": "payload.given_name",
  "last_name": "payload.family_name"
}

Users can accept or override Nova’s recommendations.

Troubleshooting Mapping Issues

  • Null / Missing Values — Use defaults or conditionals.
  • Type Mismatch — Convert strings⇄numbers⇄dates.
  • Case Sensitivity Errors — Normalize keys before mapping.
  • Object Shape Differences — Normalize upstream objects.
  • Provider Required Fields Missing — Nova will surface warnings.
  • Invalid Enum Values — Map to the provider’s valid choices.
  • ID Mismatches — Use lookup tables for cross-system consistency.

Mapping issues are the most common cause of workflow failures — SyncToScale validates mappings before runtime to reduce outages.
