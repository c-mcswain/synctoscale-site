Authentication

SyncToScale supports OAuth 2.0, API key authentication, and scoped access. This guide explains how connections are created, refreshed, validated, and monitored.

HTTP Methods at a Glance

POST Sends data to create, run, or process something
GET Retrieves information without changing anything

Most examples below show how SyncToScale receives data or manages authentication programmatically.

Overview

Every integration in SyncToScale begins with a secure, validated connection. Once authenticated, SyncToScale performs automated health checks, token refresh logic, and permission validation to ensure every workflow run has access to required resources.

POST OAuth 2.0 (Recommended)

Most modern applications use OAuth 2.0, allowing users to grant and revoke access without sharing passwords. 

POST /auth/oauth/start
{
  "provider": "google",
  "redirect_url": "https://synctoscale.me/oauth/callback"
}

Token Response

{
  "access_token": "ya29.a0AW...",
  "refresh_token": "1//0gH3...",
  "expires_in": 3600,
  "scope": "users.read calendar.readonly"
}

Tokens are encrypted at rest and refreshed automatically. If refresh fails, the connection enters a Needs Attention state.

POST API Key Authentication

Some systems use API keys instead of OAuth. SyncToScale validates keys by performing a test request to confirm scope and access. 

POST /auth/api-key
{
  "api_key": "sk_live_abc123",
  "provider": "legacy_crm"
}

GET Connection Health Checks

Connections are monitored continuously to prevent workflow failures. 

GET /connections/{id}/health
{
  "status": "healthy",
  "last_checked": "2025-01-18T12:44:00Z",
  "warnings": []
}

Error States

  • Expired Token ‚Äî Refresh failed.
  • Insufficient Permissions ‚Äî Required scopes missing.
  • Invalid Credentials ‚Äî API key or token revoked.
  • Provider Outage ‚Äî Upstream service unavailable.
  • Rate Limit Lockout ‚Äî Temporary suspension.

Best Practices

  • Use OAuth whenever possible
  • Grant minimum required scopes
  • Rotate API keys every 90 days
  • Monitor connection warnings via logs
