Authentication

SyncToScale supports OAuth 2.0, API Key authentication, and scoped access. This guide explains how connections are created, refreshed, validated, and monitored.

Overview

Every integration in SyncToScale begins with a secure, validated connection. Once authenticated, SyncToScale performs automated health checks, token refresh logic, and permission validation to ensure every workflow run has access to required resources.

OAuth 2.0 (Recommended)

Most modern applications (including HubSpot, Google Workspace, Slack, and Microsoft 365) use OAuth 2.0. This allows end-users to grant and revoke access without sharing passwords. 

POST /auth/oauth/start
{
  "provider": "google",
  "redirect_url": "https://synctoscale.me/oauth/callback"
}

Token Response

{
  "access_token": "ya29.a0AW...",
  "refresh_token": "1//0gH3...",
  "expires_in": 3600,
  "scope": "users.read calendar.readonly"
}

SyncToScale stores encrypted tokens and refreshes them automatically. If a refresh fails, the connection enters a Needs Attention state.

API Key Authentication

Some systems (like older CRM or HRIS tools) use API keys. This method is simple but less secure, as rotating credentials often requires the admin to manually replace the stored key. 

POST /auth/api-key
{
  "api_key": "sk_live_abc123",
  "provider": "legacy_crm"
}

SyncToScale validates the key by performing a lightweight test request to confirm scope and access.

Connection Status & Health Checks

Connections are monitored continuously to ensure workflows don't fail due to expired credentials. Health checks include:

  • Token validation
  • Permission scope checks
  • Rate limit monitoring
  • Endpoint availability checks
  • Recent authentication error detection
GET /connections/{id}/health

{
  "status": "healthy",
  "last_checked": "2025-01-18T12:44:00Z",
  "warnings": []
}

Error States

SyncToScale automatically detects authentication-related issues and assigns a clear state to help users troubleshoot quickly.

  • Expired Token — Refresh failed, requires reconnect.
  • Insufficient Permissions — Workflow requires scopes not granted.
  • Invalid Credentials — API key or token no longer valid.
  • Provider Outage — Upstream service unreachable.
  • Rate Limit Lockout — Temporary hold triggered by remote provider.

Best Practices

To ensure stable workflows, we recommend:

  • Using OAuth whenever possible
  • Granting the minimum required scopes for each integration
  • Rotating API keys every 90 days
  • Monitoring connection warnings via the Logs dashboard
