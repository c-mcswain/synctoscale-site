Error Handling

SyncToScale identifies, classifies, retries, and surfaces workflow errors to ensure stable automation across systems. Nova assists with diagnosing root causes and recommending fixes.

Overview

Errors can occur at any stage of a workflow: authentication, mapping, provider calls, rate limits, or internal validations. SyncToScale uses structured error handling to minimize disruption and provide clear visibility into what went wrong.

Common Error Types

400 – Bad Request

Triggered when the outgoing action payload is invalid or missing required fields. 

{
  "error": "invalid_payload",
  "details": "Missing required field: last_name"
}

401 – Unauthorized

Occurs when a connection token is expired or revoked.

403 – Forbidden

The connection lacks the necessary scopes/permissions for the operation.

404 – Not Found

The target record does not exist in the destination system.

409 – Conflict

Indicates duplicates, version conflicts, or concurrency issues.

429 – Rate Limited

Provider has throttled requests. SyncToScale applies exponential backoff and retry logic.

500 / 503 – Provider Outage

Remote system is unstable or unavailable. Automatic retries and alerts are triggered.

Retry Logic

SyncToScale retries certain transient failures automatically:

  • Rate limits
  • Network instability
  • Provider 500/503 errors
  • Temporary throttling
{
  "retry_strategy": {
    "max_attempts": 5,
    "backoff": "exponential",
    "initial_delay_ms": 2000
  }
}

Non-retryable errors (invalid mappings, permission issues, missing fields) fail immediately with actionable details.

Mapping & Payload Failures

Many workflow failures are caused by incorrect or incomplete mappings.

  • Missing required destination fields
  • Incorrect data types (string vs number vs date)
  • Invalid enum values
  • Null values where the provider requires non-null
  • Unnormalized schemas
{
  "error": "mapping_error",
  "field": "start_date",
  "expected": "YYYY-MM-DD",
  "received": "12/01/2025"
}

Nova provides suggested corrections for mapping issues.

Nova-Assisted Error Classification

Nova analyzes payloads, connection states, provider responses, and mapping structures to:

  • Identify the root cause
  • Group similar failures
  • Recommend mapping or schema changes
  • Highlight missing permissions/scopes
  • Detect provider outages or rate limits early
Nova Insight:
{
  "likely_cause": "Insufficient Permission",
  "missing_scope": "users.write",
  "recommended_fix": "Re-authenticate with required scope"
}

Error State Representation

SyncToScale uses structured error states to make troubleshooting easier.

  • Transient Failure – Retry automatically.
  • Permanent Failure – Requires user action.
  • Authentication Failure – Connection invalid.
  • Mapping Failure – Requires field correction.
  • Provider Outage – Wait until system recovers.
{
  "status": "permanent_failure",
  "error_code": "invalid_permission",
  "message": "User lacks write access to Tickets"
}

Workflow Failure Example

{
  "workflow": "new-hire-sync",
  "step": "create_user",
  "provider": "google_workspace",
  "attempts": 3,
  "status": "failed",
  "error": "rate_limit_exceeded",
  "next_retry": "2025-01-20T15:41:00Z"
}

The error log shows backoff behavior and next retry timestamp.
